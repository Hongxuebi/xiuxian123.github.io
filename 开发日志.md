### Bug修复

#### 1. 工具栏按钮全部失效（TDZ错误）
- **根因**：`备忘录UI.js` 中 `编辑内容区` 在第1272行用 `const` 声明，但第394行的 `触发输入事件()` 函数已引用它，形成暂时性死区。`绑定备忘录UI()` 执行到此处抛 `ReferenceError`，导致后续所有工具栏按钮绑定代码未执行
- **修复**：将 `const 编辑内容区` 声明移到函数顶部，删除原位置重复声明
- **教训**：`const`/`let` 声明位置必须在使用之前，不像 `var` 会提升

#### 2. 导出功能分享优先导致无文件
- **根因**：鸿蒙WebView上 `navigator.share` 返回 `true` 但实际没触发分享动作，导致 `_触发下载` 被跳过
- **修复**：移除"分享优先"逻辑，所有四个导出入口改为始终走 `_触发下载`

#### 3. 多选导出count=0、编辑页"找不到当前备忘录"
- **根因**：导出函数用了 `window._备忘录数据`（静态内存数组，页面初始化时赋值，不含IndexedDB异步加载的数据）。真实数据在 `window._备忘录数据源`（getter → `备忘录管理器.memos`）
- **修复**：4处数据源全部改为 `window._备忘录数据源 || window._备忘录数据 || []`
- **教训**：**永远用 `_备忘录数据源` 而非 `_备忘录数据`**，后者是初始化快照不是实时数据

#### 4. 卡片滑动"移动到文件夹"按钮无功能
- **修复**：从 alert 替换为实际实现

#### 5. 全局函数.js重复定义覆盖
- **问题**：`window.清空当前会话` 在全局函数.js和对话管理.js重复定义，全局函数.js的空实现覆盖了真实实现
- **待处理**：需清理全局函数.js中的重复定义

#### 6. 导出功能.js语法错误
- **问题1**：文件有多余闭合括号 `};`
- **问题2**：JSDoc注释缺少 `*/` 闭合，整个函数被当成注释
- **修复**：两处均已修复

#### 7. 筛选按钮点击后抽屉关闭
- **根因**：筛选按钮点击事件没有 `e.stopPropagation()`，事件冒泡到document触发抽屉关闭
- **修复**：备忘录事件.js 添加 `e.stopPropagation()`

### 踩坑总结

1. **TDZ（暂时性死区）**：`const`/`let` 在声明前引用直接报错，不像 `var` 提升为 `undefined`。变量声明必须在函数顶部
2. **数据源陷阱**：`_备忘录数据` 是初始化快照，`_备忘录数据源` 是实时数据。新功能永远用 `_备忘录数据源`
3. **navigator.share不可靠**：鸿蒙WebView上返回 `true` 但不执行，不能作为下载前置判断
4. **JSDoc未闭合**：缺少 `*/` 会让后续函数定义被吞噬成注释，且不报语法错误，极难排查
5. **全局函数覆盖**：多个文件定义同一个 `window.xxx`，加载顺序决定谁覆盖谁，后加载的空实现会覆盖先加载的真实实现

  2. **AI 选错了工具** —— 应该用 toggle_todo（精确切换待办状态），实际用了 update_memo（整体替换内容）
  3. **工具返回信息太少** —— AI 反复调用 read_memos 验证，挤爆轮次
- **修复**（三管齐下）：
  - **系统提示词**：增加「备忘录内容格式规范」章节，明确告知备忘录内容是 HTML，待办操作必须用 toggle_todo，禁止用 update_memo 改待办
  - **read_memos 返回**：增加 `内容格式: 'HTML'` 字段 + `关键提示: '待办状态变更必须用toggle_todo工具'`
  - **update_memo HTML 保护**：写入前检测纯文本覆盖 HTML 待办的情况，自动拒绝并引导 AI 使用 toggle_todo
  - **工具返回丰富**：update_memo/toggle_todo 返回正文摘要 + "无需再调用 read_memos 验证"
- **修改文件**：对话管理.js、API调用.js

#### 5. update_memo 工具描述强化
- **问题**：工具 description 没有明确警告纯文本覆盖会摧毁 HTML 结构
- **修复**：在 description 中强化警告：⚠️ 传入纯文本会摧毁 HTML 待办结构，系统会自动拒绝
- **修改文件**：API调用.js

### 踩坑总结

1. **tool_choice 全程 required 导致无限循环**：只在第0轮强制 required，后续轮次用 auto 让 AI 自由决定
2. **AI 不知道底层是 HTML**：read_memos 返回纯文本表示，AI 以为可以随意用纯文本覆盖 → 需要在返回中明确告知格式
3. **工具返回信息太少**：AI 会反复验证 → 返回完整信息 + "无需再验证"
4. **update_memo 没有保护机制**：纯文本覆盖 HTML 无警告 → 增加检测和自动拒绝

### 存储拆分重构

#### 背景
原方案全量备忘录以单条 JSON 字符串存 IndexedDB，全部加载到内存后渲染。数据量增长后全量序列化/反序列化性能逐渐成为瓶颈。

#### 设计方案
**写拆分**：保留全量内存缓存（运行时操作不变），持久化层改为独立文件 + 摘要索引。
- 每条备忘录存独立 `{id}.json` 文件
- 摘要索引（~50KB）写入 `_index.json`，每条只存标题/日期/预览/标签/文件夹
- 全量 1.2MB JSON → 摘要 50KB，首屏加载快 20x+
- 所有公共 API 签名不变

#### 改动
- `备忘录管理器.js`：新增 `_writeMemoFile/_deleteMemoFile/_readMemoFile` 三方法；`_saveIndex` 只写摘要（不含内容字段）；`_loadIndex` 增加旧格式自动检测迁移
- 备份原文件为 `备忘录管理器.js.bak.2026-05-03`

### Bug 修复：文件夹树点击筛选不生效

#### 问题
点击文件夹后，备忘录列表未按文件夹筛选显示，仍显示全部条目。

#### 根因
`备忘录渲染.js` 中 `渲染备忘录列表()` 函数内，`当前标签筛选` 变量的 `const` 声明位置在第 734 行（标签筛选区），但第 695 行（文件夹筛选条件）提前引用了 `!当前标签筛选`。

`const` 存在暂时性死区（TDZ），声明前访问会抛 `ReferenceError: Cannot access '当前标签筛选' before initialization`，整个 `渲染备忘录列表()` 函数执行中断——文件夹筛选逻辑根本未执行，界面保持上次渲染结果不变。

#### 修复
将 `const 当前标签筛选 = window._当前标签筛选 ? window._当前标签筛选() : null` 从第 734 行移至第 666 行，和其他状态变量（当前文件夹/当前筛选/当前日期筛选）一起声明，删除原位置重复声明。

#### 教训
- 所有状态变量的 `const`/`let` 声明必须集中放在函数开头
- 函数内变量声明位置不规范会引发 TDZ，抛错后整个函数终止运行
